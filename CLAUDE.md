# CLAUDE.md — Empathic Legal

## Project Overview

**Empathic Legal** is a two-party agreement tool that helps people negotiate contracts without confrontation. Each party privately shares their fears, needs, and priorities. An AI mediator then produces a neutral, empathic summary and suggests relevant legal protections — without ever exposing one party's raw words to the other.

This is a **sensitive-use product**. Users come here during stressful moments — business disputes, separations, partnership breakdowns. Every interaction must feel calm, safe, trustworthy, and human.

## AI Integration — Anthropic Claude API

The edge function `supabase/functions/analyze-session/index.ts` was originally built using Lovable's AI gateway with Google Gemini. **This must be migrated to the Anthropic Claude API.**

### Target setup:
- Use `claude-sonnet-4-6` as the model (best quality/cost balance for this use case)
- The API key is stored as `ANTHROPIC_API_KEY` in Supabase Edge Function secrets
- Use `anthropic-ai/sdk` via esm.sh in Deno:
  ```ts
  import Anthropic from "https://esm.sh/@anthropic-ai/sdk@0.27.0";
  ```
- Use structured output with a system prompt that enforces JSON format
- The prompt must include both parties' data but **never include raw quotes** — only field values fed in as context

### Prompt principles:
- The AI acts as a neutral mediator, not a lawyer
- Output must be in the **same language as the session context** (if context is Hebrew → respond in Hebrew)
- Output format:
  ```json
  {
    "empathic_summary": "...",
    "legal_concepts": [{ "concept": "...", "explanation": "..." }],
    "bridge_suggestions": ["...", "..."]
  }
  ```

### Environment variables required:
```
ANTHROPIC_API_KEY=sk-ant-...
SUPABASE_URL=...
SUPABASE_SERVICE_ROLE_KEY=...
```

## Supabase — Database Schema

The database is fully owned by us. Feel free to design and evolve the schema. Always write migrations for schema changes — never edit the DB manually.

### Tables

**`sessions`**
- `id` uuid PK
- `short_code` text UNIQUE — 6-char invite code (generated by `generate_short_code()` DB function)
- `context` text — description of the agreement (written by Party A, visible to Party B)
- `status` enum: `waiting_for_party_b | waiting_for_both | both_submitted | analyzed`
- `creator_user_id` uuid FK → auth.users
- `created_at`, `updated_at` timestamps

**`party_responses`**
- `id` uuid PK
- `session_id` uuid FK → sessions
- `user_id` uuid FK → auth.users
- `party_role` text: `party_a | party_b`
- `concerns` text
- `protections` text
- `priorities` text
- `desired_outcomes` text
- `submitted` boolean
- `created_at`, `updated_at` timestamps

**`ai_analyses`**
- `id` uuid PK
- `session_id` uuid FK → sessions (one-to-one)
- `empathic_summary` text
- `legal_concepts` jsonb — array of `{ concept, explanation }`
- `bridge_suggestions` jsonb — array of strings
- `created_at` timestamp

**`profiles`**
- `id` uuid PK
- `user_id` uuid FK → auth.users
- `display_name` text
- `phone` text
- `created_at`, `updated_at` timestamps

### Row Level Security (RLS)
- RLS must be enabled on all tables
- Users can only read/write their own data
- `ai_analyses` is readable by both users in the session (Party A and Party B), but **only via the service role in the edge function** — never expose raw party responses through RLS
- Party A must never be able to query Party B's `party_responses` row, and vice versa

---

## Privacy & Security — CRITICAL RULES

These rules are the core product promise. Never violate them.

1. **Party A's exact words are NEVER shown to Party B, and vice versa.** Only the AI-generated neutral summary is shared.
2. **`party_responses` rows are private to their author.** RLS must enforce this. The edge function reads both using the service role key for AI processing only.
3. **Session access via short code** (`/s/:CODE`) gives Party B access to the session `context` field only — not Party A's `party_responses`.
4. **Session access via ID** (`/session/:id`) must verify the requesting user is either Party A or Party B of that session. Never show session data to strangers.
5. When generating share links, always use the `short_code` path (`/s/CODE`), never the raw session UUID.
6. Double-check all Supabase queries for accidental data leakage — especially when doing `select('*')`.

## UX & Design Principles

This app handles emotionally charged moments. The design must reflect that.

- **Never build generic UI.** Avoid cookie-cutter SaaS patterns. Every screen should feel intentional and warm.
- **Transparency over cleverness.** Explain clearly what the app is doing at each step — especially anything involving AI or data storage.
- **Empathy in copy.** UI text should be calm, neutral, and non-judgmental. Avoid legal jargon. Avoid anything that sounds like taking sides.
- **No dark patterns.** No hidden actions, no confusing CTAs, no guilt-tripping.
- **Sensitive context = extra care.** Loading states, empty states, and error messages should feel human — not like a bank error.
- Use smooth transitions and thoughtful whitespace. The app should feel considered, not rushed.
- Hebrew copy must be reviewed for tone — auto-translated strings often sound robotic.

## Internationalization (i18n)

- UI text must never be hardcoded in JSX as plain strings. Use a translation system.
- If there is no i18n library set up yet, set one up (e.g., `react-i18next`) before adding new UI text.
- Translation keys go in `src/i18n/locales/he.json` (Hebrew) and `src/i18n/locales/en.json` (English).
- Default language is Hebrew (`he`). English is the fallback.
- The `<html>` element must have `lang` and `dir` set dynamically based on active language.
